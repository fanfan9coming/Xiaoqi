# 中医方解小程序 · 技术选型决策

> 患者拍方 → AI 方解，核心方案选型记录

## 一、项目背景

本项目是一个微信小程序，目标场景：患者就诊中医后，拍照上传中药方子，AI 辅助解析医生开方意图，解释方剂的君臣佐使配伍、药材功效、对应症状等。

### 核心难点

- **知识体系庞大**：伤寒论 113 方、金匮要略 205 方、本草纲目 1892 种药材
- **知识更新频繁**：药典改版、教材修订是常态
- **专业性极强**：不能编造条文号、不能混淆方剂出处
- **合规要求高**：不能替代医生面诊、不能给剂量调整建议

## 二、候选方案

选型阶段共评估了 3 个候选方案：

| 方案 | 核心思路 | 工具/技术栈 |
|---|---|---|
| **RAG 检索增强生成** | 通用大模型 + 中医知识库 | 扣子知识库 + Bot |
| **Skills 工具调用** | 把中医知识封装成可调用的 Skill | 扣子 Skills 市场 |
| **蒸馏（模型 / 思维）** | 大模型能力压缩 / 思维模式提取 | 自训练 / Prompt 工程 |

最终选择 **RAG 方案**，本文档记录决策过程与理由。

## 三、为什么不用 Skills 方案

### 3.1 Skills 的本质是工具封装

扣子平台的 Skills（或类似 Function Calling）核心解决的是「**让 AI 能做什么**」的问题：

- 调用天气 API → AI 能查天气
- 调用搜索 API → AI 能联网搜
- 调用日历 API → AI 能建日程

但中医方解需要的不是「工具」，而是「**专业知识**」。Skills 解决的是 Action（动作），RAG 解决的是 Knowledge（知识），两者处于不同维度。

### 3.2 知识库类任务，Skills 无法承载

把伤寒论 398 条条文、金匮要略 205 个方剂硬塞进 Skill 的 system prompt 里：

- **Token 爆炸**：伤寒论全文约 5 万字，超出单次 prompt 容量
- **检索粒度差**：「用户问太阳中风」时，没法精准只返回相关条文
- **更新困难**：条文修订要重新发布整个 Skill，迭代速度慢
- **调用成本高**：每次问答都重复加载所有知识

### 3.3 扣子商店没有现成的中医知识库

经搜索 Coze 商店 `中医` / `伤寒论` / `中药` / `方剂` / `经典` 等关键词，**全部返回空**。这意味着即使选 Skills 路线，核心知识库仍需自建——本质上就是 RAG 方案。

### 3.4 Skills 与 RAG 的正确关系

| 场景 | 推荐方案 |
|---|---|
| 查询今天天气 | Skills |
| 发送一封邮件 | Skills |
| 查某场比赛赛程 | Skills |
| 解释桂枝汤方解 | RAG |
| 分析方子的君臣佐使 | RAG |

**Skills 不是 RAG 的替代品，而是 RAG 的补充。** 本项目最终用 Skills 封装 OCR 识别、药名标准化等**功能类**任务，知识检索仍走 RAG。

## 四、为什么不用蒸馏方案

「蒸馏」在 2026 年有 2 个完全不同的含义，2 个都不适合本项目。

### 4.1 传统模型蒸馏（Knowledge Distillation）

**原理**：大模型（教师）教小模型（学生），通过匹配输出分布、思维链等方式把大模型的能力压缩到小模型里。代表：DeepSeek R1 蒸馏 1.5B、7B、70B 系列。

**为什么不适用本项目**：

| 维度 | 模型蒸馏 | 本项目需求 |
|---|---|---|
| 目标 | 模型压缩（让小模型具备部分大模型能力） | 专业知识应用（AI 解释方剂） |
| 训练成本 | 需 GPU（V100/A100）+ 大量数据 + 数周时间 | MVP 阶段无法承担 |
| 知识更新 | 改一次训练一次 | 药典改版要求天级更新 |
| 推理部署 | 需自建推理服务 | 扣子云端直接调用 |
| 错误追溯 | 模型黑盒 | RAG 能定位到具体条文 |

**核心问题**：本项目需要的不是「小模型」，而是「专业的知识」。伤寒论/金匮要略作为公开古籍，不需要「从大模型蒸馏出来」——直接读原文即可。蒸馏解决的痛点（模型太大/太慢）在 API 时代已不存在（扣子直接调云端大模型）。

### 4.2 网红的「思维蒸馏」（女娲-skill 类工具）

**原理**：把一个人的思维模式提取成 6 个维度（世界观滤镜、决策直觉、嘴替特征、雷区底线、来路、老实话），写成 prompt 让 AI 用这个人的思维方式回答问题。典型的「蒸馏毛主席」「蒸馏乔布斯」「蒸馏雷军」就是这种。

**为什么不适用本项目**：

- **古籍不是「一个人」**：伤寒论、金匮要略是「知识体系」，不是某个人的「统一思维模式」。张仲景在书里用了多种辨证框架（六经、脏腑、经络、方证），没有单一的「思维 DNA」可提炼
- **压缩反丢信息**：蒸馏的本质是「压缩」——但伤寒论 113 方已经够精炼，强行压缩成几条决策原则，会丢掉「麻黄汤 vs 桂枝汤 vs 小青龙汤」的细节差异
- **专业领域慎用**：女娲-skill 自己的文档明确说「医疗诊断、法律意见」不适合蒸馏 Agent，「它的价值在思维方式，不在专业知识」

### 4.3 蒸馏的灰色地带

2026 年 2 月，Anthropic 发文指控 DeepSeek、Moonshot（Kimi）、MiniMax 三家中国 AI 实验室对 Claude 进行工业级蒸馏攻击（1600 万 + 对话、24000 个虚假账号）。蒸馏别人模型有法律风险，蒸馏开源模型收益低——本项目作为应用层，没必要碰这条灰色路径。

### 4.4 结论

蒸馏 = **模型压缩技术** / **思维提取技巧**，**不解决「知识检索」问题**。本项目是知识应用类，不是模型优化类，不需要蒸馏。

## 五、为什么选 RAG 方案

### 5.1 RAG 完美匹配本项目需求

| 需求 | RAG 方案如何满足 |
|---|---|
| 庞大的中医知识体系 | 知识库按条文/方剂/方证切片，按需检索 |
| 知识更新灵活 | 药典改版直接换切片，零训练成本 |
| 引用具体条文 | 强制 prompt「必须写明依据《伤寒论》第 X 条」，检索结果作为证据 |
| 合规兜底 | 系统级加免责声明，不开放剂量调整建议 |
| MVP 成本 | 用扣子现成 Bot + 知识库，无需 GPU 训练 |

### 5.2 知识库设计要点

按 3 层结构切片：

- **条文级**：单条原文 + 摘要 + 主症 + 病机 + 互参条文
- **方剂级**：单方组成 + 服法 + 禁忌 + 衍生方
- **方证对应级**：方名 + 证型 + 主症 + 舌脉 + 鉴别诊断

文件格式：扣子知识库**不收 JSON**，需用 TXT。按 `[条文ID: SHL-001]` 这种标识开头，扣子自定义分段按段落切分，每条切片完整保留 metadata。

### 5.3 进阶：思维蒸馏 + RAG 的组合

虽然本项目不直接用蒸馏，但借鉴「思维蒸馏」的思路，可以**蒸馏「张仲景的辨证思维」**作为系统 prompt 的一部分：

- 六经辨证框架
- 君臣佐使配伍原则
- 方证对应、异病同治思维
- 「观其脉证，知犯何逆，随证治之」核心原则

这样 AI 不是「查条文 + 复述」，而是用张仲景的思维框架来分析方子，再调用条文佐证。RAG 解决「知识」，思维蒸馏解决「思维」，两者互补。

## 六、最终决策

**采用方案：通用大模型 + RAG 知识库（+ 思维蒸馏 prompt 可选）**

```
┌─────────────────────────────────────┐
│  基础层：扣子云端大模型（豆包/通义）   │
├─────────────────────────────────────┤
│  思维层：张仲景辨证思维 Prompt（可选） │
├─────────────────────────────────────┤
│  知识层：伤寒论 + 金匮要略 RAG 知识库  │
│          （条文/方剂/方证 3 层切片）  │
├─────────────────────────────────────┤
│  工具层：Skills 封装 OCR、药名标准化   │
└─────────────────────────────────────┘
```

**放弃的方案**：

- ❌ 全量微调（成本高、迭代慢、效果不一定更好）
- ❌ 模型蒸馏（不解决知识检索问题）
- ❌ 纯 Skills 方案（无法承载知识库类任务）

## 七、后续演进路径

1. **MVP 阶段**：完成伤寒论太阳病篇（约 100 条有效条文）切片 + 桂枝汤等核心方剂 → 跑通 Bot + 知识库链路
2. **优化阶段**：扩金匮要略全本 + 本草纲目核心药材 + 医案库 → 提升方解准确度
3. **进阶阶段**：尝试加入张仲景思维蒸馏 prompt → 提升方解深度
4. **产品化阶段**：补 OCR 模块（医疗版 OCR）、药名标准化、合规兜底文案
