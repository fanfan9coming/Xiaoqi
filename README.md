# 中医方解 AI 小程序

基于 RAG（检索增强生成）的中医处方解读系统。用户上传中医处方照片，系统通过 OCR 识别药味 → 方剂匹配 → 知识库检索 → LLM 生成方解，帮助用户理解医生开方的思路。

## 功能

- **处方 OCR 识别**：拍照识别处方上的药味（依赖外部 OCR 服务）
- **智能方剂匹配**：将药味与 378 首经典方剂进行匹配（伤寒论 92 方 + 金匮要略 143 方 + 方剂学 143 方）
- **知识库检索**：匹配到方剂后，自动检索相关条文原文和方解内容
- **AI 方解生成**：以医生实际处方为主体，对比经典方原方，分析加减变化，生成通俗易懂的解读

## 系统架构

```
用户处方照片
     │
     ▼
  OCR 识别（药味列表）
     │
     ├──────────────────┬──────────────────┐
     ▼                  ▼                  ▼
 代码节点①          代码节点②          代码节点③

     │                  │                  │

                        │

                        │
                        ▼

```

## 核心组件

### 1. 方剂匹配代码节点（×3）

三个节点结构相同，内部数据库不同：

| 节点 | 数据库 | 方剂数 | 文件大小 |
|------|--------|--------|----------|
| formula_match_node_fjx.py | 方剂学 | 143 | 25.4KB |
| formula_match_node_shanghan.py | 伤寒论 | 92 | 18.6KB |
| formula_match_node_jingui.py | 金匮要略 | 143 | 24.0KB |

**输入参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| ocr_herbs | String | 空格分隔的药名，如"熟地黄 当归 白芍 甘草" |

**输出参数**（7个，全部 String）：

| 参数 | 说明 |
|------|------|
| match_type | 匹配类型（精确匹配/超集匹配/子集匹配/模糊匹配/弱匹配/无匹配） |
| result | 最佳方名 |
| formula_name | 最佳方名 |
| category | 方剂类别（如"治风剂""和解剂"） |
| source | 出处（如"伤寒论·太阳病"） |
| composition | 组成药味 |
| candidates | Top5 候选方剂 JSON |

**匹配算法**：

- 药名标准化：两步解析（别名映射 + 炮制前缀剥离），覆盖 95%+ 临床常见异名
- 评分公式：`score = IDF加权命中和 + 0.03 × 命中数 - 0.30 × 缺失数`
- IDF 权重：394 种药味，基于方剂出现频率计算，区分度高的药味（如天麻、钩藤）权重远高于泛用药味（如甘草）
- 最少命中 2 味药才纳入候选

### 2. 合并节点

| 文件 | 大小 |
|------|------|
| formula_merge_node.py | ~2KB |

**输入参数**：

| 参数 | 来源 | 说明 |
|------|------|------|
| candidates_fjx | 方剂学节点 | candidates JSON |
| candidates_sh | 伤寒论节点 | candidates JSON |
| candidates_jg | 金匮要略节点 | candidates JSON |

**输出参数**（3个，全部 String）：

| 参数 | 说明 |
|------|------|
| candidates_brief | 简洁文本列表（每库 top 2，共 6 条），直接喂给 LLM |
| top_formula_name | 得分最高方名（用于 RAG 检索） |
| top_source | 来源库名称（用于定位知识库） |

合并节点**不做筛选判断**，只整理和排序，最终决策权交给 LLM。

### 3. RAG 知识库

| 知识库 | 内容 | 文件格式 |
|--------|------|----------|
| 方剂学 | 143 首方剂，16 个类别（解表/泻下/和解/清热/温里/补益/固涩/安神/开窍/熄风/理气/理血/祛湿/祛痰/治燥/祛风） | TXT 切片文件 |
| 伤寒论 | 92 首经方条文 + 方解 | TXT 切片文件（v0.4） |
| 金匮要略 | 143 首经方条文 + 方解 | TXT 切片文件 |

RAG 检索时使用 `top_formula_name`（方名）作为 query，命中对应条文和方解内容。

### 4. LLM 方解生成（v4 Prompt）

**输入参数**：

| 参数 | 来源 | 说明 |
|------|------|------|
| ocr_herbs | 起始节点 | 处方实际药味 |
| candidates_brief | 合并节点 | 候选方剂列表 |
| rag_result | RAG 检索节点 | 条文 + 方解内容 |

**核心逻辑**：

1. **判断主方**：从候选列表中综合判断医生最可能参考的经典方（不只看 score）
2. **对比分析**：逐味对比处方与原方（保留/减去/新增）
3. **白话解读**：用通俗语言解释处方在治什么、怎么治的
4. **免责声明**：必须附在末尾

## 扣子平台配置步骤

### 前置要求

- 扣子（Coze）平台账号
- 创建 Bot 并选择模型（推荐豆包 Pro）

### 工作流配置

1. **起始节点**：输入参数 `ocr_herbs`（String）

2. **三个方剂匹配代码节点**（可并行）：
   - 语言选择 **Python 3**（不要选 Go）
   - 分别粘贴三个 .py 文件内容
   - 配置输入参数 `ocr_herbs` 连接起始节点

3. **合并代码节点**：
   - 粘贴 formula_merge_node.py 内容
   - 输入参数连接三个匹配节点的 `candidates` 输出

4. **知识库检索节点**：
   - Query 使用合并节点的 `top_formula_name`
   - 知识库选择方剂学/伤寒论/金匮要略文档

5. **LLM 节点**：
   - System Prompt 粘贴 v4 prompt 内容
   - User Prompt 模板：
     ```
     OCR药味：{{ocr_herbs}}

     候选方剂：
     {{candidates_brief}}

     知识库检索结果：
     {{rag_result}}
     ```

### 注意事项

- 代码节点语言必须选 **Python 3**，选 Go 会报 `Expected "from" but found "async"` 错误
- 扣子代码节点大小限制约 **25KB**（已实测：25.4KB ✅ 通过，32.9KB ❌ "参数无效"）
- 三个匹配节点可并行执行，提高响应速度
- 建议将方剂学、伤寒论、金匮要略文档放入**同一个知识库**，简化 RAG 配置

## 已知问题与限制

1. **OCR 准确率**：处方手写体、模糊印刷体可能识别错误，影响匹配准确性
2. **药名异名覆盖**：目前覆盖 95%+ 常见异名，仍有少量罕见别名未收录
3. **匹配偏差**：当处方药味与多个方剂均有交叉时，可能匹配到非目标方剂（v4 prompt 已改为让 LLM 综合判断）
4. **代码节点大小限制**：扣子平台约 25KB，因此拆分为三个节点
5. **知识库内容**：方解深度取决于知识库文档质量，部分内容较简略

## 技术亮点

- **IDF 加权评分**：借鉴信息检索的 TF-IDF 思想，给高辨识力药味（天麻、钩藤等）更高权重，避免"小方剂虚高"问题
- **药名标准化两步解析**：别名映射 + 炮制前缀剥离，解决处方用名与数据库用名不一致的问题
- **三库拆分架构**：绕开扣子代码节点大小限制，同时保留完整的 378 方数据库
- **LLM 主导判断**：合并节点不做筛选，将决策权交给 LLM，避免信息丢失
